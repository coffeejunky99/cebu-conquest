Win Probability Formula

Tech Choices

Why This Stack?

Phaser.js Best-in-class 2D game engine for browser. Built-in Tiled map loader, sprite systems, and camera.
React + TypeScript Type-safe HUD components with Zustand state. Clean separation from game canvas.
Vite Instant HMR and fast builds. Works seamlessly with both React and Phaser side by side.
Socket.IO Battle calculation and state on the server — prevents client-side cheating by design.
Node.js Same JS runtime as the client. Shared data types across the whole team via socketEvents.js.
PHP + MySQL Proven REST + MAMP setup familiar to the team. JWT for stateless auth.
ZeroTier VPN Simulates LAN across machines. Lets all 4 roles dev simultaneously without a cloud server.

Performance

LOD Zoom System

ZoomConfig.js defines thresholds · ZoomManager.js switches detail levels · CameraController.js handles smooth pan/zoom

Zoom Out (min) Island view — only islandName layer visible. Minimal tile detail. High FPS.
Mid Zoom Area + district labels appear. Spot borders drawn but simplified.
Zoom In (max) Full spotName layer active. Player avatars, spot names, ownership colors all rendered.

Why LOD? The full production map is 8,000 × 9,600 px. Without level-of-detail switching, rendering all 250 × 300 tiles at once would drop FPS below 30 on most hardware.

Socket.IO + Node.js — Kei

Server State = Single Source of Truth

Clients display. The server decides.

🔒 Anti-cheat by design: Battle outcomes are computed on the server, never trusted from clients
🔄 syncState event: After every action, server broadcasts the full canonical game state to all players
📡 All clients stay in sync: District ownership, player positions, HP/AP — all from one source
⚡ Turn control: Server drives the turn clock. Clients cannot advance the turn independently
🚫 Rejected actions: Server emits ACTION_REJECTED if a move violates rules (bad adjacency, no AP, etc.)
Battle Logic

Battle Calculation — Server Only

P = A / (A + D)

P = win probability · A = attacker ATK · D = defender DEF

Step 1: Client sends ACTION_SUBMIT { type: 'attack', targetId }
Step 2: Server validates adjacency & AP. Rejects if invalid.
Step 3: calcWinRate(atk, def) → compare vs Math.random()
Step 4: Broadcast BATTLE_RESULT + syncState to all clients
Win

Attacker captures district. Owner updated in server state.

Lose

Damage based on ATK/DEF difference. District unchanged. No client-side override.

NPC

NPC Algorithm

NPCs are fully server-side agents — they act each turn automatically with no client involvement.

1
Survey adjacency
Check all neighbors in ADJACENT_DISTRICTS for empty, own, and enemy spots
2
Prioritize empty spots
Free capture with no risk → always attack empty first if available
3
Evaluate enemy spots
If P = A/(A+D) > threshold (e.g. 0.6), attempt attack on enemy district
4
Fallback: Stay
If no favorable move, recover HP +20 / AP +30 to prepare next turn
5
Broadcast result
Server emits npcUpdate → all clients render NPC movement/battle
Gotchas

Real-Time Sync Challenges

territoryUpdated — missing 'd'
❌ "territoryUpdate"  →  ✅ SERVER_EVENTS.TERRITORY_UPDATED
Client never received the event. 30 minutes of debugging a 1-character typo.
CORS: string vs array
❌ cors: { origin: ["*"] }  →  ✅ cors: { origin: "*" }
Socket.IO rejects wildcard in array form. Must be a plain string.
fromDistrictId save timing
const fromId = this.currentDistrictId;  // must be BEFORE movePlayer()
movePlayer() overwrites currentDistrictId. Capturing fromId after the call loses the origin.
Number vs String ID mismatch
ADJACENCY[11101] ✅  vs  ADJACENCY["11101"] ❌ (server uses strings, client uses numbers)
normalizeId() helper enforces Number on client. Server normalizes to String on receive.

Security

Score Validation & Security

SQL Injection Prevention

✅ All integer IDs cast: $spotId = (int)$data['territory_id']
✅ Prepared statements for all DB queries
❌ String interpolation in SQL — never used

Score Validation

✅ Final score cross-checked against server state before writing to DB
✅ JWT payload provides authoritative user_id — no spoofing
✅ XSS: React escapes output by default. PHP uses htmlspecialchars()

Defense in depth: Server computes battle results → PHP validates final scores → JWT prevents identity spoofing. Three independent layers protect game integrity.

Post-Mortem

Top 5 Bugs We Fought

🐛
React StrictMode → Phaser double init
Fix: Remove <StrictMode> from main.jsx. StrictMode intentionally double-invokes effects.
🐛
territoryUpdated typo — missing trailing 'd'
Fix: Use SERVER_EVENTS.TERRITORY_UPDATED constant. Never write event names as strings.
🐛
Socket.IO CORS: origin: ["*"] fails silently
Fix: Use origin: "*" (string). Wildcard in array form is rejected by Socket.IO with no error message.
🐛
Database connection failure — MySQL not running on host PC
Fix: Always verify XAMPP MySQL is running before testing. Confirm IP in .env matches actual network state.
🐛
Spots not clickable / not displayed on map
Fix: Verify 5-digit spot IDs are consistent across Tiled TMJ, ADJACENCY map, and DISTRICTS_MASTER. One mismatch hides everything.
Lessons

What We Learned as a Team

💡
Event name constants eliminate a whole class of bugs
Before constants: typo bugs took hours to find. After: the compiler or linter catches them instantly.
💡
Interface contracts enable true parallel development
Defining event payloads before writing code let all 4 roles build independently and integrate cleanly.
💡
Server as single source of truth solves synchronization
Once we stopped trusting client state and always re-synced from the server, race conditions disappeared.
💡
CLAUDE.md as living team contract
Keeping all rules, IDs, event names, and gotchas in one doc reduced "who changed what?" friction across 4 branches.
💡
Build the plumbing before the features
The weeks spent on ID normalization, event constants, and ADJACENCY graphs paid dividends in every sprint after.
